## 1 系统设计方案

### 1.1 总体逻辑结构设计

本平台采用 **浏览器/服务器（B/S）** 架构模式，基于现代 Web 技术栈构建，面向党政机关日常办公需求，提供覆盖"通知公告、审批流程、人员管理、日程安排、物资管理"等核心业务场景的一体化工作平台。

#### 1.1.1 总体架构分层

系统采用经典三层架构模式进行逻辑分层：

| 层级 | 功能定位 | 技术选型 |
|------|---------|---------|
| **表现层（前端）** | 用户界面呈现与交互 | React 18 + TypeScript + Tailwind CSS |
| **业务逻辑层（后端）** | 数据处理、业务规则、权限控制 | Node.js/Express（离线）/ 云端无服务器函数（在线） |
| **数据持久层（数据库）** | 结构化数据存储与管理 | MariaDB（离线部署）/ PostgreSQL（在线云端） |

#### 1.1.2 双模数据适配架构

系统的核心设计亮点为 **"双模数据适配器（Dual-Mode Data Adapter）"** 机制。通过统一的数据访问层（`dataAdapter`），前端应用可在不修改任何业务代码的前提下，无缝切换两种运行模式：

- **在线模式（Online Mode）**：通过云端数据库服务直接进行数据读写，适用于公网或云环境部署场景。前端通过标准化 SDK 与云端 PostgreSQL 数据库交互，利用行级安全策略（RLS）实现精细化权限控制。

- **离线模式（Offline Mode）**：通过本地 Node.js Express API 服务与 MariaDB 数据库交互，适用于内网隔离环境部署。所有 API 接口遵循 RESTful 规范，通过配置文件中的 `OFFLINE_MODE` 标志自动切换。

模式切换通过全局配置实现，运行时由 `window.GOV_CONFIG.OFFLINE_MODE` 参数控制。数据适配器层封装了所有数据库操作，对上层业务组件屏蔽底层差异，确保业务逻辑的一致性和可移植性。

#### 1.1.3 各模块间关系与数据流动

系统核心数据流动关系如下：

```
┌─────────────────────────────────────────────────────────┐
│                     前端应用层                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌─────────┐  │
│  │ 工作台    │  │ 审批中心  │  │ 通讯录   │  │ 管理后台 │  │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬────┘  │
│       └──────────────┴──────────────┴──────────────┘      │
│                          │                                │
│              ┌───────────┴───────────┐                    │
│              │   统一数据适配器层     │                    │
│              │   (dataAdapter.ts)    │                    │
│              └───────┬───────┬───────┘                    │
└──────────────────────┼───────┼────────────────────────────┘
                       │       │
           ┌───────────┘       └───────────┐
           ▼                               ▼
   ┌───────────────┐              ┌───────────────┐
   │  在线模式      │              │  离线模式      │
   │  云端数据库    │              │  Node.js API   │
   │  (PostgreSQL) │              │  + MariaDB     │
   └───────────────┘              └───────────────┘
```

数据流动过程涵盖以下核心链路：

1. **用户认证流**：用户登录 → 身份校验 → 本地会话存储（`localStorage`）→ 全局状态注入
2. **审批业务流**：申请提交 → 审批实例创建 → 待办分发 → 逐级审批 → 状态回写 → 业务表更新
3. **组织同步流**：外部系统推送 → 消息订阅接口 → XML 报文解析 → 数据库同步
4. **权限控制流**：用户请求 → 角色查询 → 权限矩阵匹配 → 数据范围过滤 → 结果返回

### 1.2 平台逻辑结构设计

平台逻辑结构按照使用角色划分为 **管理端** 和 **用户端** 两大子系统，二者通过统一的数据库和权限体系互联互通。

#### 1.2.1 管理端逻辑结构

管理端面向系统管理员，提供平台数据和配置的集中管理能力。管理端采用 **Tab 导航式** 单页面布局，11 个功能模块以标签页形式组织，管理员可快速切换操作区域：

```
管理后台（/admin）
├── 轮播图管理（BannerManagement）
├── 通知公告（NoticeManagement + NoticeImageManagement）
├── 食堂菜谱（MenuManagement）
├── 通讯录（ContactManagement）
├── 外出管理（AbsenceManagement）
│   ├── 出差申请管理
│   ├── 请假申请管理
│   └── 外出申请管理
├── 假期管理（LeaveBalanceManagement）
├── 办公用品（SupplyManagement）
├── 日程管理（ScheduleManagement）
├── 领导日程（LeaderScheduleManagement）
├── 审批设置（ApprovalSettings）
│   ├── 基础设置
│   ├── 表单设计
│   ├── 流程设计
│   └── 高级设置
└── 系统管理（SystemManagement）
    ├── 角色管理
    ├── 角色用户
    └── 权限管理
```

管理端的权限认证采用双重机制：
- **在线模式**：基于云端身份认证服务，结合 `user_roles` 表中的 `admin` 角色进行权限校验。
- **离线模式**：通过本地存储的管理员凭证（`adminUser`）进行身份验证，校验角色字段是否为 `admin`。

#### 1.2.2 用户端逻辑结构

用户端面向普通工作人员，提供日常办公工作台。用户端首页采用 **响应式三栏布局**，桌面端按照 2:5:3 的比例分配屏幕空间，移动端自动切换为垂直滚动布局：

```
用户工作台（/）
├── 左侧栏（20%）
│   └── 工作事项面板（WorkPanel）
│       ├── 待办列表
│       ├── 已办列表
│       └── 抄送列表
├── 中间栏（50%）
│   ├── 通知公告（NoticeList）
│   └── 应用导航（QuickLinks）
└── 右侧栏（30%）
    ├── 日程管理（SchedulePanel）
    ├── 常用链接（ExternalLinks）
    └── 食堂菜谱（CanteenMenu）
```

用户端的认证基于 `contacts` 表（通讯录人员表），用户通过手机号或账号登录后，系统将用户信息存储在 `localStorage` 的 `frontendUser` 键中，后续操作基于该身份信息进行数据筛选和权限控制。

#### 1.2.3 管理端与用户端的协同关系

管理端与用户端通过共享数据库实现业务协同：

| 协同维度 | 管理端操作 | 用户端呈现 |
|---------|-----------|-----------|
| 通知公告 | 发布、编辑、置顶、设置发布范围 | 实时查看已发布的通知，按密级过滤 |
| 食堂菜谱 | 按周维护早中晚菜品 | 查看当日及本周菜谱 |
| 审批流程 | 设计模板、配置节点和审批人 | 发起申请、处理待办、查看审批进度 |
| 日程安排 | 管理全员日程 | 查看和管理个人日程 |
| 领导日程 | 维护领导日程、配置查看权限 | 被授权用户查看领导日程 |
| 办公用品 | 管理库存、审批领用/采购 | 发起领用和采购申请 |

### 1.3 总体部署架构设计

系统支持两种部署模式，以适应不同的网络环境和安全要求。

#### 1.3.1 在线云端部署

在线模式下，系统基于云端 PaaS 平台运行，前端为静态站点部署，后端利用云端服务提供数据库、身份认证、无服务器函数等能力：

```
┌─────────────────────────────────────────────────┐
│               用户浏览器                          │
│         （React SPA 静态资源）                    │
└──────────────────────┬──────────────────────────┘
                       │ HTTPS
                       ▼
┌─────────────────────────────────────────────────┐
│              云端服务层                            │
│  ┌──────────────┐  ┌──────────────┐             │
│  │ PostgreSQL   │  │ 身份认证服务  │             │
│  │ 数据库       │  │ (Auth)       │             │
│  ├──────────────┤  ├──────────────┤             │
│  │ 行级安全策略  │  │ Edge Function│             │
│  │ (RLS)        │  │ (无服务器)    │             │
│  └──────────────┘  └──────────────┘             │
└─────────────────────────────────────────────────┘
```

#### 1.3.2 内网离线部署

内网部署模式面向政务内网隔离环境，支持国产化硬件和操作系统：

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| **操作系统** | 银河麒麟 V10（ARM64） | 支持鲲鹏 920 处理器 |
| **Web 服务** | Nginx | 反向代理、静态资源托管 |
| **应用服务** | Node.js + Express | 后端 API 服务，端口 3001 |
| **数据库** | MariaDB 10.x | 关系型数据存储，端口 3306 |

部署拓扑架构如下：

```
┌────────────────────────────────────────────────────┐
│                  Nginx (端口 80/443)                 │
│   ┌─────────────────────┬─────────────────────┐     │
│   │ 静态资源 (/)         │ API 代理 (/api/)    │     │
│   │ /opt/gov-platform/  │ → localhost:3001     │     │
│   │ web/                │                      │     │
│   └─────────────────────┴─────────────────────┘     │
└────────────────────────────┬────────────────────────┘
                             │
                             ▼
┌────────────────────────────────────────────────────┐
│           Node.js Express API (端口 3001)           │
│   ┌──────────┐  ┌──────────┐  ┌──────────────┐    │
│   │ RESTful  │  │ 文件上传  │  │ 集成接口      │    │
│   │ CRUD API │  │ (Multer) │  │ (SSO/消息订阅)│    │
│   └──────────┘  └──────────┘  └──────────────┘    │
└────────────────────────────┬────────────────────────┘
                             │
                             ▼
┌────────────────────────────────────────────────────┐
│            MariaDB (端口 3306)                       │
│   数据库：gov_platform                               │
│   字符集：utf8mb4                                    │
│   表数量：31 张核心业务表                             │
│   存储过程：deduct_leave_balance 等                   │
└────────────────────────────────────────────────────┘
```

#### 1.3.3 安全性设计

系统在安全性方面采用多层防护机制：

1. **数据访问安全**：在线模式下，每张数据表均配置了行级安全策略（RLS），通过 `has_role()` 安全函数进行细粒度权限控制，确保管理操作仅限 `admin` 角色执行，普通用户仅能查看授权范围内的数据。

2. **身份认证安全**：密码在数据库中以哈希值形式存储（`password_hash` 字段）；在线模式支持标准 OAuth 认证流程；SSO 对接采用数字签名验证机制。

3. **传输安全**：支持 HTTPS 加密传输；API 层支持 CORS 策略配置。

4. **数据密级管理**：通知公告、公文等内容支持密级标注（`security_level`），包括"公开"、"一般"、"秘密"等级别，系统根据用户密级权限过滤可见内容。

### 1.4 平台业务功能设计

平台业务功能分为管理端和用户端两大模块，管理端负责数据维护和系统配置，用户端提供日常办公操作界面。

---

#### 1.4.1 平台管理端功能

管理端为系统管理员提供全面的内容管理和系统配置能力，涵盖 11 个功能模块。

---

##### 1.4.1.1 通知公告管理

**功能概述**

通知公告管理模块为管理员提供公告信息的全生命周期管理能力，包括公告的创建、编辑、发布、置顶和删除操作，同时支持公告区域图片轮播的独立管理。

**核心功能**

| 功能项 | 说明 |
|--------|------|
| 公告创建与编辑 | 支持富文本编辑器输入公告内容，可设置标题、发布部门 |
| 密级设置 | 支持设置通知公告的安全密级（如"一般"、"秘密"等），不同密级的公告面向不同权限的用户展示 |
| 发布范围控制 | 支持设置公告的发布范围（`publish_scope`），可选择"全部可见"或按组织机构精准推送（`publish_scope_ids`） |
| 置顶功能 | 重要公告可设置为置顶（`is_pinned`），确保优先展示 |
| 发布/撤回 | 支持公告发布状态控制（`is_published`），可随时撤回已发布的公告 |
| 公告区图片管理 | 独立管理通知区域的图片轮播内容（`notice_images` 表），支持排序和启停控制 |

**数据表结构**

- `notices`：存储公告主体信息，包含标题（`title`）、内容（`content`）、部门（`department`）、密级（`security_level`）、发布范围（`publish_scope`、`publish_scope_ids`）、置顶标志（`is_pinned`）、发布状态（`is_published`）等字段。
- `notice_images`：存储公告区域轮播图片，包含图片URL（`image_url`）、标题（`title`）、排序（`sort_order`）、启用状态（`is_active`）。

**操作流程**

1. 管理员进入"通知公告"管理页面
2. 点击"新建公告"，填写标题、选择发布部门、设定密级
3. 使用富文本编辑器撰写公告正文
4. 选择发布范围（全部/指定部门）
5. 保存为草稿或直接发布
6. 已发布公告在用户端实时展示

---

##### 1.4.1.2 食堂菜谱管理

**功能概述**

食堂菜谱管理模块用于维护单位食堂的每周菜谱信息，按周一至周日组织，分早、中、晚三餐管理具体菜品。

**核心功能**

| 功能项 | 说明 |
|--------|------|
| 按周管理 | 以周为单位管理七天的菜谱，每天对应一条记录 |
| 三餐分类 | 每天菜谱按早餐（`breakfast`）、午餐（`lunch`）、晚餐（`dinner`）分类 |
| 菜品列表 | 每餐菜品以数组形式存储（`text[]`），支持灵活增删菜品 |
| 批量编辑 | 支持一次性编辑整周菜谱 |

**数据表结构**

- `canteen_menus`：以 `day_of_week`（1-7 对应周一至周日）为键组织数据，每天三个数组字段分别存储三餐菜品名称列表。

**使用场景**

管理员每周更新食堂菜谱，更新后用户端首页的"食堂菜谱"模块自动展示当日及本周菜单，方便职工提前了解餐饮安排。

---

##### 1.4.1.3 通讯录管理

**功能概述**

通讯录管理模块是平台的基础人员数据中心，负责管理组织机构树形结构和人员信息。该模块与多个业务模块深度关联，是审批流程、日程管理、权限分配等功能的人员数据来源。

**核心功能**

| 功能项 | 说明 |
|--------|------|
| 组织架构管理 | 支持多级组织树形结构管理，每个组织包含名称、简称、上级组织、层级、排序等信息 |
| 人员信息管理 | 管理人员的基本信息，包括姓名、部门、职务、手机号、办公电话、邮箱、办公地点等 |
| 在岗状态管理 | 支持多种在位状态（`contact_status`：在岗、出差、请假、外出等），与外出管理模块联动自动更新 |
| 领导标识 | 可标记人员的领导身份（`is_leader`），用于领导日程等模块的权限判断 |
| 密级设置 | 为每位人员设置安全密级（`security_level`），默认为"一般" |
| 部门主管配置 | 支持为组织设置部门负责人（`department_head_id`）和直属上级（`direct_supervisor_id`），用于审批流程中的层级审批 |
| 外部系统标识 | 支持关联外部系统用户标识（`rmsid`），用于单点登录对接 |

**数据表结构**

- `organizations`：组织机构表，支持递归树形结构（通过 `parent_id` 自引用），记录各组织的名称、层级、排序、联系方式及负责人信息。
- `contacts`：人员信息表，关联所属组织（`organization_id`），包含完整的个人信息、身份认证信息（`password_hash`、`rmsid`）及状态信息。

**操作流程**

1. 管理员构建组织架构树（添加/编辑/删除组织节点）
2. 在对应组织下添加人员信息
3. 配置人员的职务、密级、领导标识等属性
4. 设置部门负责人，用于审批流程自动匹配
5. 人员信息变更后，相关模块自动同步更新

---

##### 1.4.1.4 外出管理

**功能概述**

外出管理模块统一管理三类外出相关申请：出差申请、请假申请和外出申请。三种类型共享同一管理界面，通过子标签页切换。

**核心功能**

| 申请类型 | 功能说明 |
|---------|---------|
| **出差申请** | 管理员工出差申请，包括目的地、出差时间、交通方式、预计费用、同行人员等信息 |
| **请假申请** | 管理员工请假申请，支持 9 种假期类型（年假、病假、事假、陪产假、丧假、产假、哺乳假、婚假、调休），记录请假时长（天/小时）、事由、交接人及交接事项 |
| **外出申请** | 管理员工临时外出申请，记录外出地点、外出类型、联系电话等 |

**数据表结构**

- `absence_records`：统一存储三类申请，通过 `type` 字段（`absence_type` 枚举：`business_trip`/`leave`/`out`）区分类型。关键字段包括：
  - 通用字段：申请人（`contact_id`）、原因（`reason`）、起止时间（`start_time`/`end_time`）、状态（`status`）
  - 出差专属：目的地（`destination`）、交通方式（`transport_type`）、预计费用（`estimated_cost`）、同行人（`companions`）
  - 请假专属：假期类型（`leave_type`）、时长天数（`duration_days`）、时长小时（`duration_hours`）、交接人（`handover_person_id`）、交接事项（`handover_notes`）
  - 外出专属：外出地点（`out_location`）、外出类型（`out_type`）、联系电话（`contact_phone`）

**审批状态流转**

```
待审批(pending) → 审批中(processing) → 已批准(approved) / 已驳回(rejected)
                                      ↓
                              已取消(cancelled)（申请人可撤回）
```

**审批实例生命周期**


##### 1.4.1.10 系统权限设置

**功能概述**

系统权限设置模块实现了基于角色的访问控制（RBAC）体系，通过角色管理、角色用户分配和权限矩阵配置三个子模块，精确控制不同角色对各功能模块的操作权限和数据访问范围。

**子模块说明**

| 子模块 | 功能说明 |
|--------|---------|
| **角色管理** | 创建和维护系统角色，每个角色包含名称（`name`）、显示标签（`label`）、描述（`description`）及启用状态。系统预置角色（`is_system = true`）不可删除 |
| **角色用户** | 将用户分配到角色，一个用户可拥有多个角色（多对多关系，通过 `user_roles` 表关联） |
| **权限管理** | 配置每个角色对各功能模块的操作权限，包括增、删、改、查四种操作权限及数据范围控制 |

**权限模型**

权限矩阵通过 `role_permissions` 表实现，每条记录描述一个角色对一个功能模块的权限配置：

| 字段 | 说明 |
|------|------|
| `role` | 角色名称 |
| `module_name` | 模块标识符 |
| `module_label` | 模块显示名称 |
| `can_create` | 是否可创建 |
| `can_read` | 是否可查看 |
| `can_update` | 是否可编辑 |
| `can_delete` | 是否可删除 |
| `data_scope` | 数据范围（`personal` 仅个人 / `department` 本部门 / `all` 全部） |

**权限校验流程**

系统通过 `has_role()` 数据库函数实现高效的权限校验。该函数使用 `SECURITY DEFINER` 模式执行，避免与行级安全策略产生递归调用问题：

```sql
-- 权限校验函数
CREATE FUNCTION has_role(_user_id UUID, _role TEXT) 
RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM user_roles 
    WHERE user_id = _user_id AND role = _role
  )
$$ LANGUAGE SQL STABLE SECURITY DEFINER;
```

## 2 业务集成体系

### 2.1 机构用户同步

#### 2.1.1 功能概述

机构用户同步模块实现了本平台与上级政务服务平台的组织架构和人员信息自动同步。通过标准化的消息订阅接口，平台能够实时接收并处理来自上级系统推送的机构变更和用户变更消息，确保本平台的组织人员数据与上级系统保持一致。

#### 2.1.2 接口设计

同步接口采用 XML 报文通信协议，支持 `POST` 请求方式：

| 配置项 | 在线模式 | 离线模式 |
|--------|---------|---------|
| 接口地址 | Edge Function 端点 | `POST /api/cjgov/message-subscription` |
| 认证方式 | 无需认证（信任网络） | 无需认证（信任网络） |
| 请求格式 | XML 报文 / `application/x-www-form-urlencoded`（`request` 参数） | XML 报文 / `application/x-www-form-urlencoded`（`request` 参数） |
| 响应格式 | XML 报文 | XML 报文 |

#### 2.1.3 报文解析流程

接口实现了三层 XML 报文解析机制，用于从复杂的嵌套结构中提取有效数据：

```
第一层（Envelope）: 接收 XML 报文
    ├── type: 报文类型
    ├── signatureContent: Base64 编码内容
    │
    └── 第二层（SignatureContent）: Base64 解码
        ├── nonce: 随机标识
        ├── content: 业务内容
        │
        └── 第三层（Resources）: 解析资源列表
            ├── resource[n]: 单条变更记录
            │   ├── type: 资源类型（person/department）
            │   ├── action: 操作类型（add/update/delete）
            │   ├── no: 资源编号
            │   └── attributes: 资源属性集
            └── ...
```

#### 2.1.4 数据同步规则

| 资源类型 | 同步目标表 | 匹配字段 | 操作说明 |
|---------|-----------|---------|---------|
| `department`（组织） | `organizations` | `external_no` | 新增/更新组织信息，同步名称、上级关系、层级 |
| `person`（人员） | `contacts` | `rmsid` | 新增/更新人员信息，同步姓名、手机号、部门、职务；`resource.no` 同时映射到 `rmsid` 和 `account` 字段 |

#### 2.1.5 字符编码支持

接口内置完整的 UTF-8 字符编码支持链路：

- Base64 编码/解码采用 `TextEncoder`/`TextDecoder` 实现，确保中文字符在 Base64 转换过程中不丢失
- 数据库连接使用 `utf8mb4` 字符集，支持完整的 Unicode 字符范围
- XML 报文解析器支持中文标签名和属性值

### 2.2 单点登录对接

#### 2.2.1 功能概述

单点登录（SSO）对接模块实现了本平台与政务信任体系的统一身份认证集成。用户通过 UKey 硬件证书在客户端完成身份验证后，系统自动完成登录，无需手动输入用户名和密码。

#### 2.2.2 SSO 认证流程

SSO 认证采用"挑战-签名-验证"三步流程：

```
┌──────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  用户浏览器 │     │ 本地安全客户端  │     │  平台后端     │     │ AS 认证服务器  │
└─────┬────┘     └──────┬───────┘     └──────┬───────┘     └──────┬───────┘
      │                 │                    │                    │
      │ 1. 发起登录      │                    │                    │
      ├────────────────────────────────────► │                    │
      │                 │                    │ 2. 获取随机数挑战   │
      │                 │                    ├──────────────────► │
      │                 │                    │ ◄── 返回 challenge  │
      │ ◄───────────── 返回 challenge ────── │                    │
      │                 │                    │                    │
      │ 3. WebSocket    │                    │                    │
      │    签名请求      │                    │                    │
      ├───────────────► │                    │                    │
      │                 │ UKey 硬件签名       │                    │
      │ ◄── 返回 ticket │                    │                    │
      │                 │                    │                    │
      │ 4. 提交票据验证   │                    │                    │
      ├────────────────────────────────────► │                    │
      │                 │                    │ 5. 验证票据          │
      │                 │                    ├──────────────────► │
      │                 │                    │ ◄── 验证结果+rmsid  │
      │                 │                    │                    │
      │                 │                    │ 6. 匹配用户         │
      │                 │                    │   (rmsid→contacts) │
      │                 │                    │                    │
      │ ◄──── 返回用户信息，完成登录 ────────── │                    │
      │                 │                    │                    │
```

#### 2.2.3 各步骤技术细节

**步骤 1-2：获取随机数挑战**

- 前端调用后端接口 `POST /api/cjgov/sso/challenge`
- 后端向 AS（Authentication Server）认证服务器发起请求，获取随机数
- 后端从返回的 XML 报文中提取 `<challenge>` 节点值并返回前端

**步骤 3：本地签名**

- 前端通过 WebSocket 连接本地安全客户端（`ws://127.0.0.1:30318/`）
- 发送包含 challenge 的 XML 签名请求
- 本地安全客户端调用 UKey 硬件进行数字签名
- 解析响应 XML 中的 `<tokeninfo>` 节点获取签名票据
- 整个过程设置 15 秒超时保护

**步骤 4-5：票据验证**

- 前端将获取到的票据提交至 `POST /api/cjgov/sso/verifyTicket`
- 后端向 AS 服务器发起 `VerifyIdentity` 验证请求
- AS 服务器返回双层结构的验证结果：
  - 外层 XML 的 `<result>` 节点内容为 Base64 编码
  - Base64 解码后的内部 XML 包含状态码（`<result>`，0 为成功）和用户标识（`<rmsid>`）
  - 系统兼容 `rmsid` 可能需要额外 Base64 解码的情况（取决于 AS 服务的 `ticketFlag` 配置）

**步骤 6：用户匹配与登录**

- 后端使用提取到的 `rmsid` 在 `contacts` 表中查询匹配的用户记录
- 匹配成功后返回完整的用户信息（姓名、部门、职务、密级等）
- 前端将用户信息存储至 `localStorage`，完成登录状态建立

#### 2.2.4 兼容性设计

| 场景 | 处理方式 |
|------|---------|
| AS 服务返回格式不一致 | 后端兼容多种 XML 结构和 Base64 编码层级 |
| UKey 未插入或客户端未启动 | WebSocket 连接超时后提示用户检查硬件 |
| 内网调试模式 | 登录页面保留传统手机号/账号密码登录方式，便于开发联调 |
| 正式部署模式 | 按规范要求仅保留 SSO 登录按钮，移除传统登录表单 |
